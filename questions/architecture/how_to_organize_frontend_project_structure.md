# Как организовать структуру frontend-проекта?

---

Структура frontend-проекта должна помогать быстро находить код, разделять ответственность и понимать, какие части приложения могут зависеть друг от друга.

Нет одной универсальной структуры. На практике чаще всего выбирают подход по размеру проекта, количеству фич, роутингу, команде и требованиям к переиспользованию.

---

#### Популярные варианты

**1. По типам файлов**

Подходит для небольших проектов и простых приложений.

```txt
src/
  components/
  hooks/
  services/
  utils/
  styles/
  App.tsx
```

Плюс — легко начать. Минус — с ростом проекта код одной фичи может оказаться разбросан по разным папкам.

---

**2. По страницам или роутам**

Часто встречается в приложениях, где основная структура повторяет маршруты.

```txt
src/
  pages/
    HomePage/
      HomePage.tsx
      HomePage.module.css
    ProfilePage/
      ProfilePage.tsx
      api.ts
  shared/
    components/
    utils/
```

Плюс — удобно искать код конкретного экрана. Минус — общая логика между страницами требует аккуратного выноса в `shared`.

---

**3. По фичам**

Хорошо подходит для средних приложений, где код меняется вокруг пользовательских сценариев.

```txt
src/
  features/
    auth/
      components/
      model/
      api/
    cart/
      components/
      model/
      api/
  shared/
    ui/
    lib/
```

Плюс — код одной фичи лежит рядом. Минус — нужно договориться, что считать отдельной фичей.

---

**4. Feature-Sliced Design**

Более строгий вариант feature-based структуры с явными слоями.

```txt
src/
  app/
  pages/
  widgets/
  features/
  entities/
  shared/
```

Плюс — понятные правила зависимостей между слоями. Минус — для маленького проекта может быть слишком тяжеловесно.

---

**5. По доменам или модулям**

Часто используется в больших продуктах, где приложение разделено по бизнес-областям.

```txt
src/
  modules/
    billing/
      pages/
      components/
      services/
    users/
      pages/
      components/
      services/
  shared/
```

Плюс — структура отражает бизнес-домены. Минус — общие UI-компоненты и инфраструктуру нужно выносить осознанно.

---

**6. Layered architecture**

Подход, где приложение делится на технические или архитектурные слои.

```txt
src/
  ui/
  model/
  api/
  services/
  lib/
```

Плюс — хорошо видны роли частей приложения. Минус — без фич или доменов код пользовательского сценария может быть разбросан по слоям.

---

**7. Atomic Design**

Методология организации UI-компонентов по уровню сложности.

```txt
src/
  ui/
    atoms/
      Button/
      Input/
    molecules/
      SearchField/
    organisms/
      Header/
    templates/
      DashboardLayout/
    pages/
      DashboardPage/
```

Плюс — удобно строить дизайн-систему. Минус — Atomic Design описывает компоненты интерфейса, но не решает полностью структуру бизнес-логики, API и состояния.

---

**8. Clean Architecture или frontend в стиле Hexagonal Architecture**

Подход, где UI отделяют от бизнес-правил и инфраструктуры.

```txt
src/
  domain/
    entities/
    rules/
  application/
    use-cases/
  infrastructure/
    api/
    storage/
  presentation/
    components/
    pages/
```

Плюс — бизнес-логика меньше зависит от React, браузера или API-клиента. Минус — для обычного CRUD-интерфейса такая структура может быть избыточной.

---

**9. Monorepo или packages**

Подходит, когда есть несколько приложений или переиспользуемых библиотек.

```txt
apps/
  web/
  admin/
packages/
  ui/
  config/
  utils/
```

Плюс — удобно разделять приложения и общие пакеты. Минус — появляется дополнительная сложность сборки, версионирования и зависимостей.

---

**10. Micro-frontends**

Архитектурный подход, где большой frontend делят на независимые приложения или модули, которые могут разрабатывать и деплоить разные команды.

```txt
apps/
  shell/
  checkout/
  profile/
  catalog/
packages/
  shared-ui/
```

Плюс — команды могут развивать части продукта независимо. Минус — сложнее решать общую навигацию, состояние, стили, версии зависимостей и пользовательский опыт.

---

#### Что важно знать?

- Маленькому проекту часто достаточно структуры по типам файлов.
- Когда фич становится много, удобнее группировать код по фичам или доменам.
- Если проект строится вокруг роутинга, структура по страницам может быть самой понятной.
- FSD полезен, когда команде нужны строгие правила слоев и зависимостей.
- Layered architecture помогает явно разделять UI, модель, API и инфраструктуру.
- Atomic Design чаще используют для дизайн-систем и библиотек компонентов.
- Clean Architecture полезна, когда во frontend много независимой бизнес-логики.
- Monorepo оправдан, когда есть несколько приложений или общие пакеты.
- Micro-frontends имеют смысл для больших организаций и независимых команд, но добавляют много технической сложности.
- Общий код лучше выносить туда, где он не зависит от конкретных страниц или фич.

---

#### Частая ошибка

Копировать модную архитектуру без причины. Для небольшого проекта сложная структура может замедлить разработку сильнее, чем помочь.

---

#### Итог

Хорошая структура проекта соответствует масштабу приложения и привычкам команды. На интервью важно не назвать один "правильный" вариант, а объяснить компромиссы: простота, рост проекта, границы ответственности, переиспользование и правила зависимостей.
